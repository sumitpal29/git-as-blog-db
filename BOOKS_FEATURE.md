# Books Feature — Complete Guide

A **book** is a structured content container inside a project. Unlike flat blog posts, a book can have nested folders and files (`.md` only), each folder described by a local `index.json`. The `generate` step produces a single `map.json` that the client SDK reads — no recursive fetching needed.

## Concepts

| Term | Meaning |
|---|---|
| **Book** | A named top-level folder inside `books/` — represents one documentation set, guide, or manual |
| **Folder** | A nested directory inside a book — represents a section or chapter |
| **File** | A `.md` file inside a book or folder — represents one page |
| **index.json** | Meta config at each folder level — tracks what items (files/folders) that level contains; used internally by the CMS |
| **map.json** | Generated by the `generate` step — a single flat tree of the entire book structure; what the client SDK reads |

## File System Layout

```
projects/
  my-docs/                          ← your project
    posts/                          ← existing blog posts (unchanged)
    meta/                           ← existing generated blog meta (unchanged)
    config.json
    books/
      docs/                         ← your book (slug: "docs")
        index.json                  ← book root meta (CMS internal)
        map.json                    ← generated full tree (SDK reads this)
        home.md                     ← homepage
        installation.md
        concepts/
          index.json                ← folder-level meta (CMS internal)
          overview.md
          architecture.md
        components/
          index.json
          button.md
          input.md
          modal.md
        typography/
          index.json
          fonts.md
          scales.md
```

**URL pattern on GitHub raw CDN** (what the SDK fetches):
```
https://raw.githubusercontent.com/{owner}/{repo}/{branch}/{project}/books/{bookSlug}/{filePath}
```

Example:
```
https://raw.githubusercontent.com/sumitpal/my-site/main/my-docs/books/docs/map.json
https://raw.githubusercontent.com/sumitpal/my-site/main/my-docs/books/docs/concepts/overview.md
```

## index.json Structure

### Book root (`books/docs/index.json`)

```json
{
  "name": "Documentation",
  "slug": "docs",
  "description": "Complete reference documentation",
  "type": "book",
  "createdAt": "2026-04-25T10:00:00.000Z",
  "updatedAt": "2026-04-25T12:00:00.000Z",
  "items": [
    { "type": "file",   "name": "home",         "path": "home.md" },
    { "type": "file",   "name": "installation",  "path": "installation.md" },
    { "type": "folder", "name": "concepts",      "path": "concepts" },
    { "type": "folder", "name": "components",    "path": "components" },
    { "type": "folder", "name": "typography",    "path": "typography" }
  ]
}
```

### Nested folder (`books/docs/concepts/index.json`)

```json
{
  "name": "concepts",
  "type": "folder",
  "createdAt": "2026-04-25T10:00:00.000Z",
  "updatedAt": "2026-04-25T10:00:00.000Z",
  "items": [
    { "type": "file", "name": "overview",     "path": "concepts/overview.md" },
    { "type": "file", "name": "architecture", "path": "concepts/architecture.md" }
  ]
}
```

### Generated map (`books/docs/map.json`) — what the SDK reads

```json
{
  "name": "Documentation",
  "slug": "docs",
  "type": "book",
  "generatedAt": "2026-04-25T12:00:00.000Z",
  "tree": {
    "type": "book",
    "name": "Documentation",
    "slug": "docs",
    "items": [
      { "type": "file",   "name": "home",        "path": "home.md" },
      { "type": "file",   "name": "installation", "path": "installation.md" },
      {
        "type": "folder",
        "name": "concepts",
        "path": "concepts",
        "items": [
          { "type": "file", "name": "overview",     "path": "concepts/overview.md" },
          { "type": "file", "name": "architecture", "path": "concepts/architecture.md" }
        ]
      },
      {
        "type": "folder",
        "name": "components",
        "path": "components",
        "items": [
          { "type": "file", "name": "button", "path": "components/button.md" },
          { "type": "file", "name": "input",  "path": "components/input.md" },
          { "type": "file", "name": "modal",  "path": "components/modal.md" }
        ]
      },
      {
        "type": "folder",
        "name": "typography",
        "path": "typography",
        "items": [
          { "type": "file", "name": "fonts",  "path": "typography/fonts.md" },
          { "type": "file", "name": "scales", "path": "typography/scales.md" }
        ]
      }
    ]
  }
}
```

## Step-by-Step: Documentation Site Example

### 1. Create the project in static-cms

Open the CMS UI at `http://localhost:5173`.

- Click **New Project**
- Name: `my-docs` → creates `projects/my-docs/`

### 2. Create the book

Inside your project, go to the **Books** tab and click **New Book**.

| Field | Value |
|---|---|
| Book name (slug) | `docs` |
| Display name | `Documentation` |
| Description | `Complete reference documentation` |

Validations enforced:
- Slug must be lowercase alphanumeric with hyphens/underscores only
- No spaces, no special characters
- Max 60 characters

This creates `projects/my-docs/books/docs/index.json`.

### 3. Add folders and files

Inside the book viewer, use **New Folder** and **New File** buttons.

**Create folders:**
```
concepts/
components/
typography/
```

**Create files at root level:**
```
home.md
installation.md
```

**Create files inside each folder:**
```
concepts/overview.md
concepts/architecture.md

components/button.md
components/input.md
components/modal.md

typography/fonts.md
typography/scales.md
```

Each time you add or remove a file/folder, the `index.json` at that level (and its parent) is automatically updated by the CMS.

### 4. Write your content

Open any file in the book editor. It's a standard Markdown editor with front-matter support.

Example front-matter for `home.md`:
```markdown
---
title: Welcome to My Docs
description: Everything you need to get started
order: 1
---

# Welcome

This is the homepage of the documentation.
```

The `order` field is optional — you can use it in the client to sort items in the sidebar.

### 5. Generate and sync

Click **Generate** in the top bar — this:
1. Generates `meta/` pagination files for blog posts (existing behaviour)
2. **New:** Generates `books/docs/map.json` with the full tree

Then click **Sync** to push to GitHub.

After the push your content is live at:
```
https://raw.githubusercontent.com/{owner}/{repo}/{branch}/my-docs/books/docs/map.json
```

## Using the Client SDK in Another App

### Install

```bash
npm install blog-database-github-client
```

### Create a client

```javascript
import { createBlogClient } from 'blog-database-github-client'

const client = createBlogClient({
  repo:    'sumitpal/my-site',   // your GitHub repo
  branch:  'main',
  project: 'my-docs',           // your project folder name
})
```

### Fetch the book map (sidebar)

```javascript
// Returns the full tree from books/docs/map.json
const bookMap = await client.getBookMap('docs')

console.log(bookMap.tree.items)
// [
//   { type: 'file',   name: 'home',        path: 'home.md' },
//   { type: 'file',   name: 'installation', path: 'installation.md' },
//   { type: 'folder', name: 'concepts',     path: 'concepts', items: [...] },
//   ...
// ]
```

### Fetch a file (page content)

```javascript
// Returns { frontmatter, content } — same shape as getPost()
const page = await client.getBookFile('docs', 'concepts/overview.md')

console.log(page.frontmatter.title)  // "Overview"
console.log(page.content)            // raw markdown string
```

### Full React example — documentation page

```jsx
// DocsPage.jsx
import { useState, useEffect } from 'react'
import { createBlogClient } from 'blog-database-github-client'
import ReactMarkdown from 'react-markdown'

const client = createBlogClient({
  repo:    'sumitpal/my-site',
  branch:  'main',
  project: 'my-docs',
})

// ── Recursive sidebar tree ──────────────────────────────────────────────────

function TreeNode({ item, currentPath, onSelect }) {
  if (item.type === 'file') {
    const isActive = currentPath === item.path
    return (
      <li>
        <button
          onClick={() => onSelect(item.path)}
          style={{ fontWeight: isActive ? 'bold' : 'normal' }}
        >
          {item.frontmatter?.title ?? item.name}
        </button>
      </li>
    )
  }

  // folder
  return (
    <li>
      <span style={{ textTransform: 'capitalize' }}>{item.name}</span>
      <ul>
        {item.items.map(child => (
          <TreeNode key={child.path} item={child} currentPath={currentPath} onSelect={onSelect} />
        ))}
      </ul>
    </li>
  )
}

// ── Main docs page ──────────────────────────────────────────────────────────

export default function DocsPage() {
  const [tree,        setTree]        = useState(null)
  const [currentPath, setCurrentPath] = useState('home.md')
  const [page,        setPage]        = useState(null)
  const [loading,     setLoading]     = useState(true)

  // Load book map once on mount
  useEffect(() => {
    client.getBookMap('docs').then(map => setTree(map.tree))
  }, [])

  // Load page content whenever path changes
  useEffect(() => {
    setLoading(true)
    client
      .getBookFile('docs', currentPath)
      .then(setPage)
      .finally(() => setLoading(false))
  }, [currentPath])

  return (
    <div style={{ display: 'flex', gap: '2rem' }}>

      {/* Sidebar */}
      <nav style={{ width: 240, flexShrink: 0 }}>
        {tree ? (
          <ul>
            {tree.items.map(item => (
              <TreeNode
                key={item.path}
                item={item}
                currentPath={currentPath}
                onSelect={setCurrentPath}
              />
            ))}
          </ul>
        ) : (
          <p>Loading…</p>
        )}
      </nav>

      {/* Content */}
      <main style={{ flex: 1, maxWidth: 800 }}>
        {loading && <p>Loading…</p>}
        {!loading && page && (
          <>
            <h1>{page.frontmatter?.title}</h1>
            <ReactMarkdown>{page.content}</ReactMarkdown>
          </>
        )}
      </main>

    </div>
  )
}
```

### Next.js example — SSG documentation site

```javascript
// app/docs/[...slug]/page.js  (Next.js 14 App Router)
import { createBlogClient } from 'blog-database-github-client'
import ReactMarkdown from 'react-markdown'

const client = createBlogClient({
  repo:    'sumitpal/my-site',
  branch:  'main',
  project: 'my-docs',
})

// Pre-generate all paths at build time
export async function generateStaticParams() {
  const { tree } = await client.getBookMap('docs')

  const paths = []

  function collect(items) {
    for (const item of items) {
      if (item.type === 'file') {
        // "concepts/overview.md" → ["concepts", "overview"]
        const slug = item.path.replace(/\.md$/, '').split('/')
        paths.push({ slug })
      } else if (item.items) {
        collect(item.items)
      }
    }
  }

  collect(tree.items)
  return paths
}

export default async function DocPage({ params }) {
  const filePath = params.slug.join('/') + '.md'
  const page = await client.getBookFile('docs', filePath)

  return (
    <article>
      <h1>{page.frontmatter?.title}</h1>
      <ReactMarkdown>{page.content}</ReactMarkdown>
    </article>
  )
}

// Sidebar as a server component — fetch once, no client JS needed
export async function generateMetadata({ params }) {
  const filePath = params.slug.join('/') + '.md'
  const page = await client.getBookFile('docs', filePath)
  return {
    title: page.frontmatter?.title,
    description: page.frontmatter?.description,
  }
}
```

## Modifications to the Original Plan

The example above reveals a few things that need to change or be clarified vs. the original plan.

### 1. SDK reads `map.json`, not recursive `index.json` (important)

**Original plan:** SDK would recursively fetch each folder's `index.json`.

**Change needed:** The `generate` step produces one `books/<bookSlug>/map.json` with the full tree. The SDK reads only that file to build the sidebar — zero recursive fetching, one network request. The individual `index.json` files are strictly a CMS-internal concern and are never read by the SDK directly.

**Impact:** `generateService.js` must be extended to walk the book folder tree and write `map.json`. No change to `buildUrl` or `fetchJson`.

---

### 2. `getBookMap` and `getBookFile` — no new `buildUrl` needed

**Original concern:** Whether `buildUrl` supports the `books/` prefix.

**Confirmed:** `buildUrl` already takes any `filePath` string and appends it to `{project}/`. Passing `books/docs/map.json` or `books/docs/concepts/overview.md` works today with zero changes to `buildUrl`.

---

### 3. `getBookFile` front-matter field: `order`

The original types didn't include an `order` field. Docs and books almost always need ordered navigation (Chapter 1 before Chapter 2). The `index.json` `items` array already defines order (insertion order), and `map.json` should preserve it. No schema change needed — `order` is just an optional front-matter field authors can add; the client returns it as part of `frontmatter` via the catch-all `[key: string]: any`.

---

### 4. `getBooks()` — listing all books

The original plan included a `getBooks()` method. For this to work without a backend call, the `generate` step should also write a `books/index.json` listing all book slugs:

```json
{
  "books": [
    { "slug": "docs", "name": "Documentation" },
    { "slug": "api-reference", "name": "API Reference" }
  ],
  "generatedAt": "2026-04-25T12:00:00.000Z"
}
```

The SDK then fetches `books/index.json` for `getBooks()`.

---

### 5. File name validation — `.md` extension auto-appended by CMS

When creating a file in the CMS, users type `overview` (not `overview.md`) — the CMS appends `.md` automatically (same pattern as the existing `.json` auto-append in the data editor). Validation runs on the stem (before the extension).

---

### 6. `index.json` sync on every mutation

Every time a file or folder is created, renamed, or deleted, **all ancestor `index.json` files** up to the book root must be updated atomically. The `bookService` must handle this — it's the most important implementation detail to get right. Failures here would cause `map.json` to show stale structure.

## Validation Rules Reference

| Target | Rule | Example valid | Example invalid |
|---|---|---|---|
| Book slug | `^[a-z0-9][a-z0-9_-]*[a-z0-9]$` or single char `^[a-z0-9]$` | `docs`, `api-v2`, `my_book` | `My Docs`, `docs!`, `-docs`, `docs-` |
| Folder name | Same as book slug | `chapter-1`, `components` | `Chapter 1`, `ch@pter` |
| File stem | Same + must produce valid `.md` path | `overview`, `getting-started` | `my file`, `file.txt` |
| Max length | 60 characters | — | — |
